What is Karpenter?

Karpenter is an open-source, flexible, high-performance Kubernetes cluster autoscaler built by AWS. It’s designed to improve our application availability and cluster efficiency by rapidly launching right-sized compute resources in response to changing application load.

Key Features

  1. Improve Application Availability: Karpenter responds quickly and automatically to changes in application load, scheduling, and resource requirements. It places new workloads onto a variety of available compute resource capacity.
  2. Lower Compute Costs: Karpenter lowers cluster compute costs by looking for opportunities to remove under-utilised nodes, replace expensive nodes with cheaper alternatives, and consolidate workloads onto more efficient compute resources.
  3. Minimise Operational Overhead: Karpenter comes with a set of opinionated defaults in a single, declarative Provisioner resource which can easily be customised.
  4. No Additional Configuration Required!: Karpenter is designed to work out of the box with minimal configuration, making it easy to get started.

How It Works

Karpenter works by observing the aggregate resource requests of unscheduled pods and making decisions to launch and terminate nodes to minimize scheduling latencies and infrastructure cost. Here’s a simple breakdown of how it works:

  1. Observation: Karpenter observes the workload on your Kubernetes cluster. It watches for any pending pods that Kubernetes can’t schedule due to lack of resources.
  2. Decision Making: When an application requires more resources, Karpenter makes decisions based on the application’s specific resource requirements. It uses custom controllers and operators to manage nodes.
  3. Provisioning: Karpenter provisions nodes directly by talking to the AWS EC2 fleet API. It creates new resources and adds them to the cluster as needed.
  4. Resource Allocation: Karpenter is intelligent enough to understand your workload’s specific resource requirements and divide resources accordingly. It can handle pod requests for configurations like arm64-based instances or GPU2.
  5. Zone Awareness: Karpenter is aware of things like a pod’s volume claim Availability Zone requirements. It can correctly determine what zone to place a node in, avoiding a common issue with cluster autoscaler not making zone-aware scaling choices.

Lets look at an example:

  1. Suppose you’re running a data processing application on a Kubernetes cluster. This application processes large datasets and generates reports. The size and frequency of these datasets vary significantly, and some datasets require more resources to process than others.
  2. One day, a large dataset arrives that needs to be processed immediately. Your application creates new pods to process this dataset, but there aren’t enough resources in your cluster to schedule these pods.
  3. Karpenter observes the unscheduled pods and their resource requirements. It then talks to the AWS EC2 fleet API and provisions new nodes that match these requirements. These could be high-memory or high-CPU instances, depending on the needs of the pods.
  4. Once the new nodes are ready, the pods are scheduled on these nodes, and your application starts processing the dataset.
  5. After the dataset has been processed and the reports have been generated, the pods are terminated. Karpenter observes that the nodes are now underutilized. It decides to terminate these nodes to save costs.
  6. Before the nodes are terminated, any remaining pods are rescheduled to other nodes in the cluster.

How Karpenter Differs from Traditional Auto-Scalers

  • Traditional auto-scalers, like the Kubernetes Cluster Autoscaler, increase or decrease the size of a Kubernetes cluster based on demand. They add or remove nodes to ensure everything runs well.
  • Karpenter, on the other hand, takes a different approach. Instead of adding or removing nodes based on demand, Karpenter provisions nodes based on application requirements. This means that it can optimise resource utilisation and reduce costs. 

Lets look at an example:

Imagine you’re running a Kubernetes cluster with a traditional Cluster Autoscaler. You have a web application that experiences a sudden spike in traffic. The Cluster Autoscaler observes the increased demand and adds more nodes to the cluster to handle the load. Once the traffic spike is over, the Cluster Autoscaler removes the extra nodes to save costs. This approach is reactive - the Cluster Autoscaler reacts to changes in demand.

Now, let’s consider the same scenario but with Karpenter instead of the Cluster Autoscaler. When the traffic to your web application increases, Karpenter doesn’t just add more nodes. Instead, it looks at the specific resource requirements of the new pods (CPU, memory, etc.) that your application is trying to schedule. Karpenter then talks to the cloud provider’s API (e.g., AWS EC2) and provisions nodes that exactly match these requirements. This could be a mix of different types of nodes - some with more CPU, some with more memory, depending on what the pods need. This approach is proactive and application-aware - Karpenter provisions nodes based on application requirements, not just demand.

Key Concepts

Provisioners:

A Provisioner in Karpenter is a custom resource that specifies the configuration for node provisioning. It sets constraints on the nodes that can be created by Karpenter and the pods that can run on those nodes. Here are some key points about Provisioners:
Taints: Provisioners can define taints to limit the pods that can run on nodes Karpenter creates.
Zones, Instance Types, and Architectures: Provisioners can limit node creation to certain zones, instance types, and computer architectures.
Node Expiration: Provisioners can set defaults for node expiration1.
Multiple Provisioners: You can have multiple Provisioners configured on the same cluster.

Example:

Based of the pod's resource requirements, the below provisioner selects the CHEAPEST node from "t3" or "m6" instance families, and then, will only select micro, small or medium instance types.

apiVersion: karpenter.sh/v1alpha5
kind: Provisioner
metadata:
  name: default-provisioner
spec:
  requirements:
    # Include general purpose instance families
    - key: karpenter.k8s.aws/instance-family
      operator: In
      values: [t3, m6]
    # Use only smaller instance sizes
    - key: karpenter.k8s.aws/instance-size
      operator: In
      values: [micro, small, medium]
  ttlSecondsUntilExpired: 604800 # expire nodes after 7 days (in seconds) = 7 * 60 * 60 * 24
  ttlSecondsAfterEmpty: 60 # scale down nodes after 60 seconds without workloads (excluding daemons)
  providerRef:
    name: default-node-template


AWS-Node-Templates

The AWSNodeTemplate is a custom resource in Karpenter that enables configuration of AWS specific settings. Each provisioner must reference an AWSNodeTemplate using spec.providerRef.

Here are some key points about AWSNodeTemplate:
Subnet and Security Group Selection: The AWSNodeTemplate discovers subnets and security groups using AWS tags.
Instance Profile, AMI, and UserData: The AWSNodeTemplate allows you to specify an instance profile, AMI family, and UserData.
Tags: The AWSNodeTemplate allows you to propagate tags to underlying EC2 resources.
Block Device Mappings and Detailed Monitoring: The AWSNodeTemplate allows you to configure storage devices for the instance and detailed monitoring.

Example: 

The below AwsNodeTemplate specifies that Provisioners can run nodes in AWS Subnets that have the tag "karpenter.sh/discovery: non-prod" and use Security-Groups that also have the tag "karpenter.sh/discovery: non-prod". And, an EBS volume of 20GB is attached to each node.

apiVersion: karpenter.k8s.aws/v1alpha1
kind: AWSNodeTemplate
metadata:
  name: default-node-template
spec:
  amiFamily: AL2
  subnetSelector:
    karpenter.sh/discovery: non-prod
  securityGroupSelector:
    karpenter.sh/discovery: non-prod
  blockDeviceMappings:
    - deviceName: /dev/xvdb
      ebs:
        volumeSize: 20Gi
        volumeType: gp3
        deleteOnTermination: true


How Karpenter is configured in project-z

In project-z, Karptenter is installed using Terraform

Karpenter requires a runtime environment and worker nodes to run on. It’s installed within Fargate, aligning with our serverless commitment.

Default AwsNodeTemplates and Provisioners

We created the following defaults:

AwsNodeTemplates

  • an AwsNodeTemplate that use AmazonLinux2 AMIs
  • an AwsNodeTemplate that use BottleRocket AMIs

Provisioner

  • an Provisioner that use AmazonLinux2 AwsNodeTemplate (has the highest priority)
  • an Provisioner that use BottleRocket AwsNodeTemplate

If one were to deploy a pod into EKS, the default AmazonLinux2 Provisioner will be used to provision a node to run that pod.

Custom Provisioner for EKS add-ons, 3rd-party and monitoring applications

We created a custom provisioner - eks-addons-amz-lnx-prov - for creating one single node for running ALL EKS add-ons, 3rd-party and monitoring applications.

Key features of this provisioner:

  • creates a label on the node: workload: eks-addons
  • taints the node with: eks-addons:true:NoSchedule

All EKS add-ons, 3rd-party and monitoring applications have the following configured to ensure that they are deployed on the node created by the custom provisioner - eks-addons-amz-lnx-prov

  • nodeSelector: workload: eks-addons
  • tolerations for: eks-addons:true:NoSchedule

Ensuring that the Daemonsets created by EKS add-ons, 3rd-party and monitoring applications only run on Karpenter provisioned nodes

Fargate does not support Daemonsets. So, to ensure that any Daemonsets created by EKS add-ons, 3rd-party and monitoring applications ONLY run on Karpetner provisioner nodes, we need to add the following "Affinity" rule to the Daemonsets:

affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: eks.amazonaws.com/compute-type
          operator: NotIn
          values:
          - fargate

Provisioners for project-z Pilot Deployment: one provisioner per test environment

To save costs, we decided to run a single, entire, project-z Pilot deployment test environment on a single, dedicated, node.

To do this, we dynamically create one provisioner per test environment, during test environment creation.

The code for dynamic provisioner creation, using Ansible, is here: 

src/ansible/roles/create-karpenter-provisioner/defaults/main.yml
---
tmp_dir: "/tmp/karpenter"

aws_node_template_name: amazon-linux-nt

mode_settings:
  performance:
    instance_type: m6a.4xlarge
    limits:
      cpu: 16000m
      memory: 64Gi
  cost-saving:
    instance_type: t3a.2xlarge
    limits:
      cpu: 8000m
      memory: 32Gi


src/ansible/roles/create-karpenter-provisioner/tasks/main.yml
---
- name: "Print test env name"
  debug:
    msg: "Creating Karpenter Provisioner for Test Env - {{ env_name }}"

- name: "Create a tmp dir to store files"
  file:
    path: "{{ tmp_dir }}"
    state: directory

- name: "Render the template file and copy it to tmp dir"
  ansible.builtin.template:
    src: "templates/provisioner.yaml.j2"
    dest: "{{ tmp_dir }}"
    mode: "0644"

- name: "Print Ansible rendered manifest template"
  debug:
    msg: "{{ lookup('file', '{{ tmp_dir }}/provisioner.yaml.j2') }}"

- name: Create the Karpenter Provisioner
  kubernetes.core.k8s:
    state: present
    src: "{{ tmp_dir }}/provisioner.yaml.j2"

- name: "Delete the tmp dir"
  file:
    path: "{{ tmp_dir }}"
    state: absent



src/ansible/roles/create-karpenter-provisioner/templates/provisioner.yaml.j2
---
#jinja2: trim_blocks: False

apiVersion: karpenter.sh/v1alpha5
kind: Provisioner
metadata:
  name: "{{ env_name }}-{{ deployment_mode }}-prov"
spec:
  labels:
    env_name: {{ env_name }}
    deployment_mode: {{ deployment_mode }}
  providerRef:
    name: {{ aws_node_template_name }}
  requirements:
    - key: kubernetes.io/arch
      operator: In
      values:
        - amd64
    - key: kubernetes.io/os
      operator: In
      values:
        - linux
    - key: karpenter.sh/capacity-type
      operator: In
      values:
        - on-demand
    - key: node.kubernetes.io/instance-type
      operator: In
      values:
        - {{ mode_settings[deployment_mode].instance_type }}   
  limits:
    resources:
      {{ mode_settings[deployment_mode].limits | to_yaml }}
  taints:
    - effect: NoSchedule
      key: env_name
      value: {{ env_name }}
    - effect: NoSchedule
      key: deployment_mode
      value: {{ deployment_mode }}
  ttlSecondsAfterEmpty: 60
  ttlSecondsUntilExpired: 1209600


How do we ensure that one test environment runs on a single, dedicated, node?

When an test environment is created, we dynamically create a dedicated provisioner for that test environment.

The provisioner has the following configured:

  • labels the node: environment: <name-of-test-env>
  • taints the node with:environment: <name-of-test-env>NoSchedule

Then, when deploying all the project-z Pilot applications for the test environment, we configure the following for each of them:

  • nodeSelector: environment: <name-of-test-env>
  • tolerations for: environment: <name-of-test-env>:NoSchedule
