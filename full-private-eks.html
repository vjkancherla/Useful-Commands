An overview of the components

EKS Control Plane

  • The EKS control plane consists of control plane nodes that run the Kubernetes software, such as etcd and the Kubernetes API server.
  • It is a managed service provided by AWS.

AWS Fargate

  • AWS Fargate is a serverless technology that provides on-demand, right-sized compute capacity for containers
  • There are no servers to manage with Fargate.
  • EKS can be used to run pods on Fargate without manually provisioning any servers.
  • Using Fargate, Kubernetes pods run with just the compute capacity they request

Karpenter

  • Karpenter is an open-source tool by AWS
  • Karpenter automates the provisioning and scaling of Kubernetes clusters' worker nodes
  • Karpenter optimizes resource utilization based on workload demands.
  • When is Pod is ready to be scheduled, Karpenter proivisions a right-sized node based on the Pod's resource requirements

Core-Services:

Core Services encompass essential Kubernetes applications crucial for the operation and security of our applications. The key services include:

  • CoreDNS: Facilitates domain name resolution and service discovery within the Kubernetes cluster.
  • Kube-Proxy: Manages network communication between nodes and handles service load balancing.
  • EBS-CSI-Driver: Enables dynamic provisioning of Amazon Elastic Block Store (EBS) volumes for persistent storage within the cluster.
  • Calico: Implements networking and network security, allowing for efficient communication between containers across nodes.
  • Calico API Server: Provides a management interface for Calico, enhancing control and visibility into networking policies.
  • Karpenter: as already discussed


A NOTE about why Calico and Calico API server are used

  • Our AWS VPC's CIDR range only provides a very limited range of IPs, which is not sufficient for our Applications
  • Calico is used to create a virtual, extended CIDR range to be used by Application Pods
  • Calico contributes to network security by implementing and enforcing Network Policies within the cluster.
  • The Calico API server is used to create custom Calico CRD objects, which are in-turn used to create the expanded CIDR range and network policies

"hostNetwork"

  • In EKS, you can set the "hostNetwork" field in a pod's specification to true to enable the host network mode. 
  • This means the containers within the pod share the network namespace with the underlying EC2 instances in the node group, effectively using the host's network stack.
  • When using "hostNetwork" the IP addresses for Pods are allocated from within AWS VPC CIDR
  • The Control Plane and all the core-service Pods on Fargate use "hostNetwork"

"Calico Network"

  • If "hostNetwork" field in a pod's specification to not set to true, then the Pods use the Calico's virtual network
  • The IP addresses for Pods are allocated from within Calico's CIDR


An IMPORTANT NOTE about connectivity between hostNetwork and Calico Network

  • Pods on hostNetwork and Calico Network ARE NOT CONFIGURED to communicate with each other
  • IMPORTANT: The K8s API Server (part of Control Plane, on hostNetwork) will not be able to connect to any WebHooks running on Calico Network

Third (3rd) Party add-ons and Observability add-ons

These Kubernetes applications deliver crucial functionality for the operation and monitoring of both the EKS cluster and applications.

  • EFS-CSI-Driver: Facilitates dynamic provisioning of Amazon Elastic File System (EFS) volumes, enhancing storage capabilities for applications within the cluster.
  • Ingress Nginx: Manages external access to services within the cluster, acting as an HTTP and reverse proxy server for efficient and secure traffic routing.
  • External Secrets Operator: Enables the synchronization of secrets between AWS SecretsManager and project-z EKS.
  • Kube-cost: Offers insights into resource usage and costs, aiding in the analysis and optimization of resource consumption within the Kubernetes environment.
  • Metrics-Server: Collects resource metrics from the cluster and provides real-time data, supporting accurate monitoring and scaling decisions.
  • Prometheus: A powerful monitoring and alerting toolkit that gathers and stores time-series data, empowering users with advanced monitoring and alerting capabilities.
  • Grafana: Works in tandem with Prometheus, providing a visualization platform for monitoring metrics and creating dashboards to gain insights into the cluster's performance.
  • Loki: A log aggregation system designed for Kubernetes, offering efficient log storage and retrieval for enhanced observability and troubleshooting.
  • Cert-Manager: X.509 certificate controller. It will obtain certificates from a variety of Issuers, both popular public Issuers as well as private Issuers, and ensure the certificates are valid and up-to-date, and will attempt to renew certificates at a configured time before expiry.
  • KEDA: KEDA is a Kubernetes-based Event Driven Autoscaling component. It provides event driven scale for any container running in Kubernetes

BottleRocket AMIs

  • Bottlerocket is optimized for running containers with high security isolation. 
  • BottleRocket AMIs are designed with a minimalistic approach, significantly reducing the attack surface compared to standard EKS AMIs. 
  • The host OS is extremely minimal, it does not come with bash, an interpreter, ssh, or anything beyond the system basics needed to run containers.
  • It uses an immutable root filesystem, and secure boot and limited shell access. You aren't intended to run or install things directly on the host at all.
  • By leveraging BottleRocket AMIs, we are aligning ourselves with industry best practices and strengthening our ability to meet compliance standards. 


Architecture Overview: Control Plane, AWS Fargate, and (Karpenter provisioned) Nodes

Brief Description

  • Single Control Plane
  • AWS Fargate for running core services
  • (Karpenter provisioned) Nodes for running:
        - 3rd add-ons required for providing extended functionality 
        - Cluster monitoring applications
        - project-z Pilot DeploymentApplication workloads
       
  • Fully private cluster, with no in-bound or out-bound internet access
  • Access to K8s API Server only from MC VPN, and peered VPCs (Jenkins Tooling VPC)

Serverless Architecture

  • We’ve adopted a serverless strategy for our EKS cluster, thereby eliminating the necessity for worker node management.
  • This serverless approach is made possible through the combined use of Fargate and Karpenter.
  • As Karpenter requires a runtime environment, it’s installed within Fargate, aligning with our serverless commitment.
  • Following the installation and configuration of Karpenter, all subsequent applications are deployed on nodes provisioned by Karpenter, allowing us to maintain our serverless infrastructure and leverage the benefits of Karpenter simultaneously.
  • Note that, in conjunction with Karpenter, we’ve also installed its dependent services on Fargate. These include CoreDNS, EBS-CSI-Driver, and kube-proxy. This ensures a seamless operation of our serverless infrastructure, as these services work in tandem with Karpenter to provide a robust and efficient environment.


EKS Platform and Terraform

Things to remember

  • Full private cluster with no outbound internet access
  • No "static" or "managed" worker nodes. Using Fargate and Karpenter instead for dynamic node provisioning
  • The Control Plane (K8s API Server) on "hostNetwork" (see earlier description), which is the default network created by AWS-VPC Container Network Interface (CNI)
  • The core-services workloads running on Fargate also run on "hostNetwork" , which is the default network created by AWS-VPC Container Network Interface (CNI)
  • Workloads that run on Karpenter nodes run on Calico Network, which is a custom, overlay, network created by us, using Calico CNI
  • IMPORTANT: workloads running on"hostNetwork" and Calico Network CANNOT communicate with each other
  • Calico CNI is used for:
    - applying K8s network policies
    - providing an expanded CIDR range for pods (removing the restriction applied by VPC's CIDR)
  • Container images stored are in private ECR repositories
  • Helm charts are stored in JFrog Artifactory
  • The following Core-service/Add-ons are installed, which are running on Fargate:
            - Kube-proxy
            - Coredns
            - Ebs-csi-driver
            - Calico
            - Calico api-server
            - Karpenter
  • The following 3rd party add ons are installed, which are running on Karpenter provisioned nodes:
     - Efs-csi-driver
    - Ingress-nginx
    - Cert-manager
    - External Secrets Operator
    - Metrics-server
    - Kubecost
    - Grafana
    - Loki
    - Prometheus
  • The following Terraform modules are used:

A description of what every layer does

000-eks

  • EKS cluster, named - project-z-non-prod, created within private subnets
  • Kubernetes v1.27 in being used
  • No public endpoint for API access
  • Only a private endpoint for API access is enabled
  •  Access to private endpoint is restricted to Company-Z VPN

Points to Remember:

  • When an EKS cluster is created, a default Security Group (SG), referred to as "primary" SG, is automatically created (by AWS)
  • The SG's name will be of the format - "eks-cluster-sg-{cluster-name}-{UID}"
  • This "primary" SG cannot be updated, when creating the cluster. 
  • We created an additional SG with all our ingress rules and added it to the cluster as an "additonal SG" to the cluster

005-eks-fargate

  • We use Fargate profiles to declare which Pods run on Fargate. Fargate profiles use "selectors" to identify pods to schedule on Fargate. Each selector must contain a namespace. The selector can also include labels.
  • We created three Fargate profiles, with selectors configured to target pods in the following namespaces:
        - kube-system
        - karpenter
        - calico-apiserver
      NOTE: For each Fargate profile selector, we have also specified that it should only look for pods with label - "farget = true"
      

Points to Remember:

  • Fargate profiles use the EKS cluster's primary security group automatically. We cannot specify a custom SG for Fargate profiles
  • Nodes created by Fargate will have AWS-VPC CNI and Fluentbit (for log aggregation) add-ons automatically installed
  • All workloads that run on Fargate will run on the "hostNetwork", created by AWS-VPC CNI

IMPORTANT NOTE ABOUT AWS-AUTH CONFIG-MAP: 

  • When a Fargate profile is created a K8s configmap called "aws-auth" is created in kube-system namespace automatically. 
  • The configmap contains the IAM permissions for allowing nodes created by Fargate Profiles to talk to the K8s API-Server.
  • DO NOT EDIT THE CONFIGMAP MANUALLY. IF THE CONFIGMAP IS NOT FORMATTED CORRECTLY, YOU WILL SEE ISSUES WHEN FARGATE TRIES TO SCHEDULE PODS.
        See: https://repost.aws/knowledge-center/fargate-troubleshoot-profile-creation

010-eks-core-addons

Core Services encompass essential Kubernetes applications crucial for the operation and security of our applications. The key services include:

  • CoreDNS: Facilitates domain name resolution and service discovery within the Kubernetes cluster.
  • Kube-Proxy: Manages network communication between nodes and handles service load balancing.
  • EBS-CSI-Driver: Enables dynamic provisioning of Amazon Elastic Block Store (EBS) volumes for persistent storage within the cluster.
  • Calico: Implements networking and network security, allowing for efficient communication between containers across nodes.
  • Calico API Server: Provides a management interface for Calico, enhancing control and visibility into networking policies.
  • Karpenter: as already discussed


Points to Remember:    

  • Note that we only "enable" the Kube-proxy add-on here. This will create a daemonset now and will launch a Kube-proxy pod on each Karpenter managed/launched node later on
  • As previously mentioned, all the above add-ons run on the"hostNetwork", created by aws-vpc CNI. This means that the K8s API server can talk to all these add-ons.
  • We create an Iam-Role-for-Service-Account (IRSA) for Ebs-csi-driver's controller to allow it to dynamically create EBS volumes for PV and PVCs
  • A "StorageClass" K8s object is created, which uses the EBS-CSI-Driver, to dynamically create EBS backed Persistent Volume objects.
  • We create an Calico "IPPool" object which specifies the EXPANDED CIDR block - 192.168.0.0/16 - to use for the IPs of pods deployed on the Calico network
  • Custom Helm charts were created for Calico and Calico API-Server
  • One of the KEY configuration settings in our custom Calico helm chart is - nodeselector. This is to ensure that calico-node daemonset pod only run on Karpenter nodes, ignoring all Fargate nodes
  • All helm charts are stored in JFrog Artifactory. To read/download the helm charts, we need to use Artifactory credentials.
  • The JFrog Artifactory credentials are stored in AWS Secrets Manager. In Terraform, we read the credentials from AWS Secrets Manager and use them when deploying helm charts.
  • The container images, referenced by helm charts, are stored in AWS ECR. So, to recap - the helm charts are stored in Artifactory and the container images, referenced in the helm charts, are stored in AWS ECR.
  • We are using AWS Blueprint Terraform modules for creating the add-ons. These modules will automatically create the MINIMUM permissions (IAM and IRSA) to allow the add-ons to function.

015-eks-karpenter

Here, we install and configure Karpenter auto-scaler to run on Fargate

  • When an application Pod needs to be scheduled, Karpenter dynamically calculates and launches a right-sized EC2 instance-type, based on the Pod's memory and CPU requirements.
  • We create a Security Group (SG) to be used by exclusively Karpenter nodes. This SG has ingress rules for allowing traffic:

      - from Control Planes's Primary SG
      - between Karpenter nodes themselves

  • The SG also has a tag - "karpenter.sh/discovery" = "{{EKS-Cluster-Name}}", which is used by Karpenter controller for discovering the SGs that it can add to nodes that it creates 
  • We tag all the VPC Subnets that we want to use with Karpenter with - "karpenter.sh/discovery" = "{{EKS-Cluster-Name}}"
  • We add a VPC endpoint for "SSM" to allow private access
  • We create a "AWSNodeTemplate" Karpenter object to define:

    - the subnets we want deploy ours nodes to
    - the SGs to use with the nodes
    - BOTTLE-ROCKET AMI  Amazon Linux2 AMI
    - IAM role to use for nodes
    - EBS volumes    

  • We create a "Provisioner" Karpenter object to define:
    - the pool of AWS EC2 instance types that Karpenter can choose an instance from
    - whether node consolidation is enabled (currently not enabled)
    - period of time to wait until a node can be terminated when unused

Points to Remember:    

  • We create an Iam-Role-for-Service-Account (IRSA) for Karpenter's controller to allow it to create EC2 instances, with encrypted EBS volumes
  • The Karpenter helm chart is stored in JFrog Artifactory. To read/download the helm charts, we need to use Artifactory credentials.
  • The JFrog Artifactory credentials are stored in AWS Secrets Manager. In Terraform, we read the credentials from AWS Secrets Manager and use them when deploying helm charts.
  • The Karpenter container images are stored in AWS ECR. So, to recap - the helm charts are stored in Artifactory and the container images, referenced in the helm charts, are stored in AWS ECR.

IMPORTANT POINTS ABOUT DEPLOYING DAEMONSETS ON KARPENTER

  • When we are deploying Daemonsets, we need to ensure that we use "node affinity" settings to ensure that the Daemonsets only run on Karpenter provisioned nodes, and ignore Fargate nodes (which do not support Daemonsets)
  • We need to add the following node affinity rule to the Daemoset:
    affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
        - matchExpressions:
          - key: eks.amazonaws.com/compute-type
            operator: NotIn
            values:
            - fargate
